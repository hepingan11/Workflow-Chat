# Digital Employee OS v0.1 开发启动计划

本文档基于 `MVP-plan.md` 整理，目标是把项目从产品想法落到可执行的第一版开发计划。

v0.1 的核心目标不是做完整的数字员工平台，而是先跑通一条最小闭环：

```text
用户创建目标 -> 系统拆解任务 -> Agent 执行任务 -> 工具调用(可以调用Dify、n8n、Coze等平台的工作流api) -> 人工审批 -> 进度汇报 -> 审计留痕
```

## 1. v0.1 产品边界

### 1.1 一句话目标

做一个面向开发者的数字研发员工平台。用户给出一个研发目标，系统能拆成可追踪任务，并由 Agent 调用本地工具持续推进。

### 1.2 首个 Demo 场景

Demo 名称：

```text
AI 开源项目维护员工
```

用户输入示例：

```text
帮我维护这个 GitHub 仓库：
1. 检查项目结构
2. 找到一个适合修复的问题
3. 制定任务计划
4. 修改代码
5. 运行测试
6. 输出完成汇报
```

v0.1 不强制自动创建 PR，可以先在本地完成修改、测试和汇报。GitHub PR 能力可以放到 v0.2 或 v0.3。

### 1.3 v0.1 必须包含

- Goal 创建与查看
- Task Graph 生成与状态流转
- Agent Runtime 基础执行循环
- Tool Registry 基础工具注册
- 本地文件系统工具
- Shell 工具
- 人工审批机制
- 进度汇报
- 执行事件日志
- 简单 Web 控制台或 API 调试入口

### 1.4 v0.1 暂不包含

- 多租户团队空间
- 插件市场
- 完整 MCP 生态
- Telegram / Slack / QQ 消息网关
- 复杂长期记忆训练
- 复杂多 Agent 组织结构
- Dify / n8n / Coze 工作流集成
- 生产级沙箱隔离
- 自动支付、发邮件、转账等高风险外部动作

## 4. v0.1 数据表设计

### 4.1 设计原则

- Goal 是用户想达成的最终目标
- Task 是可执行、可追踪、可恢复的工作单元
- Agent 是执行任务的角色
- Tool Call 是 Agent 调用外部能力的记录
- Approval 是人类介入点
- Event 是完整审计日志
- Artifact 是任务执行产生的文件、报告、代码变更或其他产物

### 4.2 枚举定义

#### goal_status

```text
draft
planning
running
waiting_approval
completed
failed
cancelled
```

#### task_status

```text
pending
ready
running
waiting_approval
blocked
completed
failed
cancelled
```

#### run_status

```text
running
completed
failed
cancelled
```

#### risk_level

```text
low
medium
high
forbidden
```

#### approval_status

```text
pending
approved
rejected
expired
cancelled
```

#### event_type

```text
goal.created

goal.started
goal.completed
goal.failed

task.blocked

tool.failed
approval.requested
approval.approved
approval.rejected
report.created
artifact.created
```

### 4.3 `goals`

用途：记录用户创建的目标。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid pk | Goal ID |
| `title` | varchar(255) | 目标标题 |
| `description` | text | 目标描述 |
| `status` | varchar(32) | 目标状态 |
| `success_criteria` | jsonb | 成功标准 |
| `deadline_at` | timestamptz nullable | 截止时间 |
| `created_by` | varchar(128) nullable | 创建人 |
| `metadata` | jsonb | 扩展字段 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |
| `completed_at` | timestamptz nullable | 完成时间 |

建议索引：

```sql
create index idx_goals_status on goals(status);
create index idx_goals_created_at on goals(created_at desc);
```

### 4.4 `agents`

用途：记录平台里的数字员工角色。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid pk | Agent ID |
| `key` | varchar(64) unique | 机器可读标识，如 `dev_agent` |
| `name` | varchar(128) | 展示名称 |
| `role` | varchar(128) | 角色类型 |
| `description` | text | 角色说明 |
| `system_prompt` | text | 角色系统提示词 |
| `tool_permissions` | jsonb | 可用工具列表 |
| `approval_policy` | jsonb | 审批策略 |
| `is_active` | boolean | 是否启用 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |

建议初始 Agent：

```text
planner_agent
dev_agent
review_agent
```

### 4.5 `tasks`

用途：记录 Goal 拆解出的任务图节点。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | uuid pk | Task ID |
| `goal_id` | uuid fk | 所属 Goal |
| `parent_task_id` | uuid nullable | 父任务 |
| `title` | varchar(255) | 任务标题 |
| `description` | text | 任务描述 |
| `status` | varchar(32) | 任务状态 |
| `priority` | integer | 优先级，数字越大越优先 |
| `owner_agent_id` | uuid fk nullable | 负责 Agent |
| `depends_on` | jsonb | 依赖任务 ID 列表 |
| `risk_level` | varchar(32) | 风险等级 |
| `approval_required` | boolean | 是否需要审批 |
| `expected_output` | text | 期望产出 |
| `result_summary` | text nullable | 结果摘要 |
| `retry_count` | integer | 已重试次数 |
| `max_retries` | integer | 最大重试次数 |
| `metadata` | jsonb | 扩展字段 |
| `created_at` | timestamptz | 创建时间 |
| `updated_at` | timestamptz | 更新时间 |
| `started_at` | timestamptz nullable | 开始时间 |
| `completed_at` | timestamptz nullable | 完成时间 |

建议索引：

```sql
create index idx_tasks_goal_id on tasks(goal_id);
create index idx_tasks_status on tasks(status);
create index idx_tasks_owner_agent_id on tasks(owner_agent_id);
create index idx_tasks_priority on tasks(priority desc);
```

说明：

`depends_on` v0.1 可以先用 `jsonb` 保存 uuid 列表。等任务图能力复杂后，再拆成 `task_dependencies` 表。

## 5. 核心状态流转

### 5.1 Goal 状态流转

```text
draft -> planning -> running -> completed
                         |
                         -> waiting_approval
                         |
                         -> failed
                         |
                         -> cancelled
```

### 5.2 Task 状态流转

```text
pending -> ready -> running -> completed
                    |
                    -> waiting_approval
                    |
                    -> blocked
                    |
                    -> failed
                    |
                    -> cancelled
```

### 5.3 审批流转

```text
pending -> approved
        -> rejected
        -> expired
        -> cancelled
```

## 8. 第一版 README 应该强调什么

README 第一屏建议表达：

```text
Not another workflow builder.
Not another chatbot.
An open-source runtime for goal-driven digital employees.
```

核心卖点：

- Goal-driven：用户给目标，不是只发 prompt
- Long-running：任务可以持续推进
- Task Graph：任务可追踪、可恢复
- Human-in-the-loop：重大操作需要审批
- Auditable：所有动作有事件日志
- Tool-native：工作流、MCP、Shell、GitHub 都只是工具
- Verifiable：任务完成需要产物和证据

## 9. 关键工程原则

### 9.1 先做状态管理，再做智能

数字员工平台的核心不是模型调用，而是：

```text
目标状态、任务状态、工具调用状态、审批状态、执行证据
```

LLM 输出可以不稳定，但状态系统必须稳定。

### 9.2 所有 Agent 行为都要留痕

任何关键动作都应该写入 `events`：

- 谁做的
- 做了什么
- 为什么做
- 输入是什么
- 输出是什么
- 是否需要审批
- 最终结果是什么

### 9.3 工具调用先保守

v0.1 宁愿工具少一点，也要保证：

- 可观察
- 可审批
- 可回放
- 可解释

### 9.4 不要过早做通用平台

第一版只服务一个场景：

```text
开发者数字员工
```

等这个场景跑通后，再扩展到研究、运营、销售、客服等角色。
